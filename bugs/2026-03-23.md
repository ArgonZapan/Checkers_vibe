## Bugs

### BUG-001: aiMove zawsze wybiera pierwszy legalny ruch — ignoruje predykcję sieci neuronowej
- **Severity:** krytyczny
- **Plik:** `server/index.js:497`
- **Status:** ✅ FIXED (commit e7d035f)
- **Opis:** Funkcja `predict()` w `model.js` zwraca `{ move: selectedMove }` gdzie `selectedMove` to pełny obiekt ruchu (np. `{from: [2,1], to: [3,0], index: 5}`). Ale `aiMove()` w `server/index.js` traktowała `prediction.move` jako indeks liczbowy:
  ```js
  const moveIndex = prediction.move;
  let selectedMove = legalMoves[moveIndex] || legalMoves[0];
  ```
  Ponieważ `moveIndex` jest obiektem (nie liczbą), `legalMoves[object]` zawsze zwraca `undefined`, więc fallback `legalMoves[0]` był zawsze używany. **AI w trybie PvAI grało pierwszy dostępny ruch co turę, kompletnie ignorując predykcje sieci neuronowej.**
- **Fix:** Użyć `prediction.move` bezpośrednio jako obiekt ruchu, z walidacją czy jest w legalMoves:
  ```js
  let selectedMove = prediction.move;
  if (!selectedMove || !legalMoves.some(m => m.from === selectedMove.from && m.to === selectedMove.to)) {
    selectedMove = legalMoves[Math.floor(Math.random() * legalMoves.length)];
  }
  ```

### BUG-002: Board.jsx areEqual nie porównuje prop `captures`
- **Severity:** ważny
- **Plik:** `client/src/components/Board.jsx:233`
- **Opis:** Niestandardowa funkcja porównująca `areEqual` (używana z `React.memo`) nie sprawdza prop `captures`. Jeśli `captures` zmieni się bez zmiany innych props, komponent się nie przerysuje. W praktyce nie powoduje buga bo `captures` z self-play jest zawsze puste, ale kod jest niespójny z designem.
- **Fix:** Dodać do `areEqual`:
  ```js
  if (prevProps.captures?.length !== nextProps.captures?.length) return false;
  ```

### BUG-003: setSpeed WebSocket walidacja niezgodna z clampingiem
- **Severity:** ważny
- **Plik:** `server/index.js:378-383`
- **Opis:** Handler `setSpeed` waliduje zakres 0-60000ms, ale potem clampuje do 0-10000ms:
  ```js
  if (typeof ms !== 'number' || ms < 0 || ms > 60000 || Number.isNaN(ms)) { ... }
  const clamped = Math.max(0, Math.min(ms, 10000));
  ```
  Wartości 10001-60000 przechodzą walidację ale są cicho clampowane. Użytkownik nie dostaje feedbacku że jego wartość została zmieniona.
- **Fix:** Albo zrównać zakres walidacji z clampingiem (0-10000), albo odrzucić wartości > 10000 z błędem.

### BUG-004: turnColor ternary w trainer._playGame jest misleading
- **Severity:** kosmetyczny
- **Plik:** `server/ai/trainer.js` (wewnątrz `_playGame`, blok emit state)
- **Opis:** Kod:
  ```js
  const turnColor = newState.turn === 1 || newState.turn === 'white' ? 'white' : 'black';
  ```
  `newState.turn` jest zawsze stringiem ("white"/"black") bo `getGameState()` konwertuje przez `turnToColor`. Warunek `=== 1` nigdy nie jest prawdą. Kod działa poprawnie ale jest misleading — sugeruje że `turn` może być liczbą.
- **Fix:** Uprościć do `const turnColor = newState.turn;` lub dodać assert.

### BUG-005: redundantna obsługa "no legal moves" w trainer._playGame
- **Severity:** kosmetyczny
- **Plik:** `server/ai/trainer.js` (wewnątrz `_playGame`, blok `if (!legalMoves || legalMoves.length === 0)`)
- **Opis:** Po sprawdzeniu `gameOver` (z C++ engine), jest dodatkowy blok obsługujący przypadek "no legal moves" z własną logiką gameOver. Ta logika duplikuje standardową obsługę gameOver powyżej. Jeśli engine nie zgłasza gameOver mimo braku ruchów, ten blok działa jako safety net — ale kod jest redundantny i trudniejszy w utrzymaniu.
- **Fix:** Zaufać engine'owi że zwraca poprawny `gameOver` lub zunifikować logikę.

### BUG-006: gamePlayed inkrementowane podwójnie na /reset
- **Severity:** kosmetyczny
- **Plik:** `engine/src/server.cpp:237`

- **Fix:** Usunąć `gamesPlayed++` z `/api/game/reset`.
