# Data Leak Check — Checkers_vibe (2026-03-24)

**Scanner:** data-leak-checker sub-agent  
**Scope:** `/opt/Checkers_vibe` — server, client, proxy, C++ engine  
**Result:** 5 findings (2 low, 2 informational, 1 medium)

---

### LEAK-001: API token accepted via URL query string
- **Severity:** Medium
- **Location:** `server/index.js:115`
- **What is leaked:** The API authentication token (`API_TOKEN`) is accepted via `?token=<value>` query parameter
- **Where exposed:** Browser history, HTTP proxy logs, CDN/reverse-proxy access logs, referer headers to third-party resources
- **Code:**
  ```js
  const provided = req.headers['authorization']?.replace(/^Bearer\s+/i, '')
    || req.query?.token;
  ```
- **Impact:** If a user opens a URL like `/api/ai/params?token=SECRET` in their browser, the token gets logged in browser history and potentially leaked via Referer header. Server-side access logs (nginx, etc.) will also contain the token in plaintext.
- **Recommendation:** Remove `req.query?.token` fallback. Tokens should only be accepted via the `Authorization: Bearer` header.

### LEAK-002: Full model architecture broadcast to all WS clients
- **Severity:** Low
- **Location:** `server/index.js:547` (on connection), `server/index.js:824` (on setParams)
- **What is leaked:** Full neural network architecture parameters (layers, neurons, activation, lr, batchSize, dropout, gamma, bufferSize, epochs, all reward weights) are broadcast to ALL connected WebSocket clients via `io.emit('paramsUpdate', { modelParams: { ...trainer.modelParams }, ... })`
- **Where exposed:** Any connected WebSocket client (no per-socket auth check on the broadcast)
- **Code:**
  ```js
  // Line 547 — sent to every new WS connection (no auth check)
  socket.emit('paramsUpdate', {
    modelParams: { ...trainer.modelParams },
    ...
  });
  // Line 824 — broadcast to ALL clients after setParams
  io.emit('paramsUpdate', { modelParams: { ...trainer.modelParams }, ... });
  ```
- **Impact:** Any visitor can enumerate the full internal model architecture, training hyperparameters, and reward shaping weights. This aids reverse-engineering of the AI strategy.
- **Recommendation:** The `paramsUpdate` broadcast should be gated behind `wsAuth()`. Unauthenticated clients receive game state only, not model internals.

### LEAK-003: Unauthenticated REST endpoints expose internal AI state
- **Severity:** Low
- **Location:** `server/index.js:142-153` (`GET /api/ai/info`), `server/index.js:313` (`GET /api/ai/stats`)
- **What is leaked:** Network sizes (white/black), epsilon values, games played, buffer size, running status, win/loss/draw statistics, avg round time, last 10 round times, total time
- **Where exposed:** Any HTTP client (no `requireApiToken` middleware)
- **Code:**
  ```js
  // No auth middleware:
  app.get('/api/ai/info', (_req, res) => {
    const status = trainer.getStatus();
    res.json({
      modelWhite: trainer.networkSizeWhite,
      modelBlack: trainer.networkSizeBlack,
      epsilonWhite: trainer.epsilonWhite,
      epsilonBlack: trainer.epsilonBlack,
      gamesPlayed: status.stats.gamesPlayed,
      bufferSize: status.bufferSize,
      running: status.running
    });
  });

  app.get('/api/ai/stats', (_req, res) => {
    res.json(trainer.getStatus().stats);
  });
  ```
- **Impact:** Attackers can enumerate training progress, model architecture hints (via network size names), exploration parameters, and performance metrics without authentication.
- **Recommendation:** Apply `requireApiToken` to `/api/ai/info` and `/api/ai/stats`, or at minimum redact `bufferSize` and `networkSize` from unauthenticated responses.

### LEAK-004: Proxy logs request URLs (potential body data leak in edge cases)
- **Severity:** Informational
- **Location:** `server/proxy.js:31-32`
- **What is leaked:** Full request URL including path is logged for non-GET requests proxied to the C++ backend
- **Where exposed:** Server-side console logs only
- **Code:**
  ```js
  if (req.method !== 'GET') {
    console.log(`[Proxy] → ${req.method} ${req.url} → ${CPP_TARGET}`);
  }
  ```
- **Impact:** Low — server-side only. However, if the C++ engine ever returns error responses with request details embedded, or if someone crafts a URL with encoded sensitive data in the path, it would appear in logs. Currently benign.
- **Recommendation:** Consider redacting query parameters from logged URLs.

### LEAK-005: Server-side console logs expose model params and internal state
- **Severity:** Informational
- **Location:** `server/ai/trainer.js:474`, `server/ai/trainer.js:680`, `server/index.js:1155`
- **What is leaked:**
  - Full `modelParams` object (architecture details) logged on every params update
  - Epsilon values, games played, running state logged on state load
  - Unhandled rejection handler logs error stack traces
- **Where exposed:** Server-side stdout/stderr only (not to clients)
- **Code:**
  ```js
  // trainer.js:474
  console.log('[SelfPlay] Model params updated:', this.modelParams);
  // trainer.js:680
  console.log(`[SelfPlay] Loaded state: ${this.stats.gamesPlayed} games played, εW=${this.epsilonWhite}, εB=${this.epsilonBlack}, running=${this.running}`);
  // index.js:1155
  console.error('[UnhandledRejection]', reason instanceof Error ? reason.stack || reason.message : reason);
  ```
- **Impact:** If server logs are shipped to a centralized logging system, or if the server is compromised, internal architecture and state are exposed. Not a direct client-facing leak.
- **Recommendation:** Acceptable for development. For production, consider structured logging with redacted model internals.

## Positive Findings (No Leaks)

The following areas were checked and found secure:

- **Error responses:** All HTTP error handlers return sanitized, generic messages. No stack traces, internal paths, or C++ engine details leak to clients. The `err.message` is logged server-side only; clients receive `{ error: '...' }` with fixed strings.
- **WebSocket error events:** All `socket.emit('error', ...)` calls send generic or controlled messages. No raw error objects or stack traces.
- **ErrorBoundary (React):** Stores error in state but renders only generic "Coś poszło nie tak" — no error details shown to users.
- **HTTP security headers:** X-Powered-By disabled (both `app.disable()` and middleware `removeHeader()`), CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy all set correctly.
- **C++ engine error handling:** `server.cpp` catches `json::parse_error`, `json::type_error`, and generic `std::exception` — all return `{ "error": "..." }` with fixed strings. No leak of internal state.
- **C++ engine binding:** Bound to `127.0.0.1:8080` — not externally accessible.
- **Proxy error handling:** Returns `{ error: 'C++ backend unavailable' }` on proxy errors — no upstream error details leaked.
- **WS auth (SEC #157):** `startSelfPlay`, `stopSelfPlay`, `setParams`, `setSpeed`, `setSpeedMode`, `reset`, `restart` all check `wsAuth()` before executing.
- **Rate limiting:** Per-IP rate limiting with cleanup interval prevents enumeration attacks.
- **CSP:** Strict Content-Security-Policy prevents inline scripts, restricts connect-src to self + wss:.
