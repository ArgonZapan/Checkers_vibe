## New Findings

### BUG-019: Minimax pawn captures restricted to forward-only (rules violation)
- **Severity:** krytyczny
- **Lokalizacja:** server/ai/minimax.js:118-124
- **Opis:** W `generateLegalMoves()` (minimax), bicia pionków są ograniczone do kierunków "do przodu" (white: dr>0, black: dr<0). To jest BŁĄD — w warunkach międzynarodowych pionki BICIA wykonują we WSZYSTKICH 4 kierunkach diagonalnych. Pionki chodzą tylko do przodu, ale biją również do tyłu. Kod pomija ~50% możliwych bić pionków, co znacząco osłabia silnik minimax.
```js
// BŁĄD: pionki biją tylko do przodu
if (isWhiteTurn && dr < 0) continue;  // white nie bije w górę
if (!isWhiteTurn && dr > 0) continue; // black nie bije w dół
```
To samo dotyczy `_extendCapture` (linia 232-238) — multi-bicie pionków również ograniczone do forward.
- **Sugerowany fix:** Usunąć filtry kierunkowe z bloku `!isKing` w `generateLegalMoves()` i `_extendCapture()`. Pionki powinny móc bić we wszystkich 4 diagonalach:
```js
// Poprawione: pionki biją we wszystkich kierunkach
for (const [dr, dc] of [[-1, -1], [-1, 1], [1, -1], [1, 1]]) {
  // BEZ filtra isWhiteTurn/dr — pionki biją do przodu i do tyłu
```

### BUG-020: Model lock window — dispose during training possible
- **Severity:** ważny
- **Lokalizacja:** server/ai/trainer.js:755-760 vs 1015-1020
- **Opis:** W `_playGame()`, model lock jest zaciągany na czas predykcji i zwalniany tuż po:
```js
// 755: predykcja z lockiem
modelRelease = await this.acquireModelLock();
const pred = await predict(model, boardArray, movesWithIndex, turn);
// 760: lock zwolniony ZANIM trening
} finally { if (modelRelease) modelRelease(); }
```
Następnie trening dzieje się 20+ iteracji później (linia 1015) z NOWYM zaciągnięciem locka. Między tymi dwoma punktami, `setParams()` lub `resetModel()` (wywoływane przez WebSocket) może wywołać `_replaceModel()` → `old.dispose()`. `_replaceModel()` czeka na poprzedni lock, ale lock był już zwolniony na linii 760. Jeśli `setParams` zdarzy się w tym oknie, stary model zostanie zDisposed, a `this.modelWhite/black` wskazuje na nowy model — trening użyje nowego modelu (OK dzięki `paramsVersion` check), ale próbki z predykcji starego modela będą trenować nowy model. To jest niekonsekwentne (samples were generated by old model, trained on new model).
- **Sugerowany fix:** Albo utrzymywać model lock przez cały cykl _playGame (od predykcji do treningu), albo dodać weryfikację modelu w treningu:
```js
trainRelease = await this.acquireModelLock();
// Weryfikuj że model nie zmienił się od predykcji
if (this.paramsVersion !== playGameVersion) { trainRelease(); break; }
```
(To już jest sprawdzane na linii 1009, ale lock jest zaciągany PO tej weryfikacji — istnieje teoretyczny race window)

### BUG-021: Auto-save nie restoreuje dirty flag po nieudanym zapisie
- **Severity:** ważny
- **Lokalizacja:** server/index.js:838-868
- **Opis:** W `_autoSaveInterval`, `trainer.dirty = false` jest ustawiane PRZED zakończeniem asynchronicznego zapisu (linia 847). Jeśli zapis się nie powiedzie (np. pełny dysk), `catch` loguje błąd ale NIE restoreuje `dirty = true`. Dane z ostatniego cyklu treningowego zostają utracone — następny cykl auto-save widzi `dirty = false` i pomija zapis.
```js
trainer.dirty = false; // ← snapshot PRZED save
// ... async save operations ...
catch (err) {
  console.error('[AutoSave] Save error:', err.message);
  // ← BRAK: trainer.dirty = true;
}
```
- **Sugerowany fix:** Dodać `trainer.dirty = true;` w bloku `catch`:
```js
catch (err) {
  trainer.dirty = true; // restore — next cycle retries
  console.error('[AutoSave] Save error:', err.message);
}
```

### BUG-022: C++ draw detection threshold niezgodny z zasadami międzynarodowymi
- **Severity:** ważny
- **Lokalizacja:** engine/src/engine.cpp:73-77
- **Opis:** `getResult()` sprawdza remis gdy `movesWithoutCapture_ >= 40` (40 pół-ruchów = 20 pełnych ruchów). Komentarz mówi "20 pełnych ruchów", ale w warunkach międzynarodowych (polish draughts / FMJD), remis wymaga 25 kolejnych pełnych ruchów przez KAŻDEGO gracza bez bicia (łącznie 50 pół-ruchów). Threshold 40 jest zbyt niski — gry kończą się remisem za wcześnie, zmniejszając jakość treningowych próbek.
```cpp
if (movesWithoutCapture_ >= 40) { // 40 pół-ruchów = 20 pełnych ruchów
    return DRAW; // powinno być >= 50 (25 pełnych ruchów)
}
```
- **Sugerowany fix:** Zmienić threshold na 50 (lub zaimplementować per-player counting zgodnie z zasadami FMJD):
```cpp
if (movesWithoutCapture_ >= 50) { // 50 half-moves = 25 full moves per player
    return DRAW;
}
```

### BUG-023: startGame akceptuje nieznane tryby bez błędu
- **Severity:** niski
- **Lokalizacja:** server/index.js:506
- **Opis:** Gdy klient wysyła `startGame` z nieprawidłowym trybem (np. `{ mode: "hacker" }`), kod cicho ustawia tryb na `'pvai'` zamiast odesłać błąd:
```js
const validModes = ['pvai', 'pvp', 'aivai'];
socket.gameMode = validModes.includes(mode) ? mode : 'pvai'; // ← cichy fallback
```
Klient może myśleć że gra w wybranym trybie, a serwer ustawił pvai. Brak informacji zwrotnej.
- **Sugerowany fix:** Odesłać błąd lub ostrzeżenie:
```js
if (!validModes.includes(mode)) {
  socket.emit('error', { message: `Invalid mode '${mode}' — using pvai` });
}
socket.gameMode = validModes.includes(mode) ? mode : 'pvai';
```

### BUG-024: multiCapture rollback nie przywraca wasPromotion state
- **Severity:** niski
- **Lokalizacja:** engine/src/movegen.cpp:~140-155 (pawn capture path)
- **Opis:** W `multiCapture`, gdy pionek awansuje na króla (linia ~140, `becameKing = true`), rekurencyjne wywołanie kontynuuje bicie jako król. Po powrocie z rekurencji, rollback przywraca `board.whitePieces` / `board.whiteKings` z zapisanych wartości (linia ~150), co jest poprawne. Ale zmienna `becameKing` jest przekazywana do rekurencji i tam używana do iteracji ALL_DIRS — po rollbacku pionek wraca do bycia pionkiem, ale w międzyczasie mógł wykonać bicia królewskie (sliding captures). Te bicia są cofane przez rollback, więc wynik jest poprawny. Jednakże kod jest subtelny i podatny na regresje.
- **Sugerowany fix:** Dodać komentarz wyjaśniający, że rollback jest poprawny bo przywraca całą planszę. Ewentualnie rozważyć głęboką kopię planszy zamiast mutacji in-place dla czytelności (koszt: wydajność).

### BUG-025: PvP io.emit wysyła stan do WSZYSTKICH socketów (włącznie z ruchem)
- **Severity:** niski
- **Lokalizacja:** server/index.js:441-445
- **Opis:** W trybie PvP, `handleMove` wysyła stan do wykonawcy ruchu via `socket.emit` (linia ~430, PvAI path). Ale w PvP używa `io.emit` (linia 442) co wysyła do WSZYSTKICH podłączonych klientów — włączając tych którzy nie grają. Obserwatorzy widzą stan gry z legalMoves (choć filtr na kliencie ignoruje legalMoves z eventu state). Dodatkowo, gracz który wykonał ruch dostaje podwójną aktualizację stanu (raz z handleMove przed AI, raz z io.emit).
- **Sugerowany fix:** W PvP, rozważyć dedykowany "room" per gra i emitować tylko do graczy w pokoju. Na razie niski priorytet — funkcjonalnie działa, ale marnuje pasmo.

## Edge Cases Found (not bugs, but worth documenting)

### EDGE-001: Trainer cppFetch zwraca Response object zamiast JSON
- **Lokalizacja:** server/ai/trainer.js:118-128
- **Opis:** `trainer.cppFetch` zwraca surowy `Response` (z `res.ok` checkiem — skorygowane w poprzednim raporcie). Wszyscy wywołujący robią `response.json()` osobno. To działa, ale jest niekonsekwentne z `server/index.js:cppFetch` który zwraca sparsowany JSON. Rozważ unifikację.

### EDGE-002: Race condition w PvP — ruchy na starym stanie
- **Lokalizacja:** server/index.js (move queue per-socket)
- **Opis:** W PvP, kolejka ruchów jest per-socket. Gracz A i Gracz B mają osobne kolejki. Jeśli A wyśle ruch, a B wyśle swój ruch PRZED otrzymaniem aktualizacji stanu, ruch B może być oparty na stanie sprzed ruchu A. Silnik C++ odrzuci go (illegal move), ale UX jest zły — gracz widzi "legalny" ruch odrzucony.

### EDGE-003: RESET w trakcie _playGame — sample pollution
- **Lokalizacja:** server/ai/trainer.js:~700-1040 + server/index.js:882-898
- **Opis:** Gdy klient wywołuje `reset` (WebSocket), `trainer.resetModel()` czyści buffer i modele, ale jeśli `_playGame` jest w trakcie, bieżąca iteracja kontynuuje aż do `while (this.running)`. Reset ustawia `running = false`, więc pętla zatrzyma się po bieżącej grze. Ale próbki wygenerowane przed resetem zostaną dodane do nowo wyczyszczonego bufora — `resetModel` czyści buffer PRZED zakończeniem _playGame, więc `_playGame` doda próbki do pustego bufora. Następnie `resetModel` kasuje plik modelu z dysku. Na koniec, `saveState` w _playGame nadpisuje stan resetu. Efekt: statystyki resetowane przez resetModel zostają nadpisane przez saveState z _playGame.

## Verdict

Najpoważniejszy bug to **BUG-019** — minimax engine nie potrafi bić do tyłu pionkami, co łamie zasady warunków międzynarodowych i drastycznie osłabia grę silnika minimax w porównaniu do DQN. To jest regresja funkcjonalna — każde ustawienie `strategy: 'minimax'` dla jednej ze stron produkuje niepoprawne zachowanie.

BUG-021 (dirty flag) to latent data loss risk — pojawia się tylko przy błędach dysku, ale kiedy występuje, traci dane bez ostrzeżenia.

Pozostałe bugi to edge cases i UX issues o niskim do średnim wpływie.
