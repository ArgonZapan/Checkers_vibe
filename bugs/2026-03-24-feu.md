# Frontend Audit Report — Checkers_vibe
**Date:** 2026-03-24  
**Auditor:** Jarvis Horner (subagent)  
**Scope:** `client/` — HTML, React components, CSS, accessibility, error handling

---

### FE-001: Missing Content-Security-Policy meta tag
- **Severity:** Medium (security)
- **Location:** `client/index.html`
- **Description:** No `Content-Security-Policy` meta tag or HTTP header. The app loads scripts via Vite (`/src/main.jsx`), uses WebSocket connections to `/`, and has inline styles. Without CSP, any XSS vulnerability allows full script injection.
- **Suggested fix:** Add `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; connect-src 'self' ws: wss:; img-src 'self' data:;">` and tune as needed.

### FE-002: No favicon
- **Severity:** Low (cosmetic)
- **Location:** `client/index.html`
- **Description:** No `<link rel="icon">` and no `public/` directory. Browsers will 404 on `/favicon.ico`.
- **Suggested fix:** Add a favicon `<link>` or create `public/favicon.ico`.

### FE-003: `selfPlayActive` missing from ref-sync dependency array
- **Severity:** Medium (state sync)
- **Location:** `client/src/App.jsx`, line ~104
- **Description:** The `useEffect` that syncs state to refs (`boardRef`, `turnRef`, etc.) does not include `selfPlayActive` in its dependency array. However, `selfPlayActiveRef` is updated inside the effect body only when `selfPlayActive` changes. This means if `selfPlayActive` changes but other listed deps don't, the effect might not run (React may skip it), leaving `selfPlayActiveRef.current` stale. The `handleToggleSelfplay` callback reads `selfPlayActiveRef.current`, so it could toggle the wrong direction.
- **Suggested fix:** Add `selfPlayActive` to the dependency array: `}, [board, turn, selected, legalMoves, mode, gameOver, selfPlayActive]);`

### FE-004: Params changes silently ignored outside AI-vs-AI mode
- **Severity:** Medium (UX)
- **Location:** `client/src/App.jsx` (`handleParamsChange`) + `client/src/components/ParamsPanel.jsx`
- **Description:** `handleParamsChange` only emits `setParams` to the server when `modeRef.current === 'aivai'`. In PvAI mode, sliders and dropdowns move and values update in local state, but have zero effect on gameplay. There's no warning or indicator that these controls are read-only in PvAI mode. The user may spend time tuning parameters that do nothing.
- **Suggested fix:** Either disable the params panel in PvAI mode, show a clear "Only active in AI vs AI" banner, or show a toast when user tries to change params in PvAI.

### FE-005: GameTimer component exists but is never rendered
- **Severity:** Low (dead code)
- **Location:** `client/src/components/GameTimer.jsx`
- **Description:** `GameTimer.jsx` defines a timer component with start/stop/reset logic, but it's never imported or rendered in `App.jsx` or any other component.
- **Suggested fix:** Either integrate GameTimer into the game UI (e.g., near the board) or remove the dead file to reduce confusion.

### FE-006: Board cell + piece both have onClick — potential double-fire in SVG
- **Severity:** Low (edge case)
- **Location:** `client/src/components/Board.jsx` (rendering loop)
- **Description:** Each dark square cell `<rect>` has an `onClick={() => onCellClick(row, col)}`, and each piece `<g>` on that cell also has `onClick={() => onCellClick(row, col)}`. In SVG hit-testing, only the topmost element receives the click (the piece `<g>`), so in practice this doesn't double-fire. However, the piece `<g>` click handler is redundant since the cell rect already handles it, and if SVG rendering order changes, it could cause confusion.
- **Suggested fix:** Remove the `onClick` from the piece `<g>` (the cell rect underneath handles the click), or remove it from the rect and keep it only on the piece (with a fallback for empty cells).

### FE-007: Excessive tab stops on the checkers board (accessibility)
- **Severity:** Medium (accessibility/UX)
- **Location:** `client/src/components/Board.jsx`
- **Description:** Every dark square cell and every piece is `tabIndex={0}`. With 32 dark squares and up to 24 pieces, the board alone generates up to 56 tab stops. A keyboard user pressing Tab has to cycle through all of them to get past the board. Standard practice for game boards is to make the entire board a single tab stop with arrow-key navigation (roving tabindex pattern).
- **Suggested fix:** Implement roving tabindex: make the board container a single `tabIndex={0}`, track a focused cell in state, and use arrow keys to move focus within the board. Show a visual focus ring on the currently focused cell.

### FE-008: `inert` attribute missing on game-over overlay — background still interactive
- **Severity:** Medium (accessibility)
- **Location:** `client/src/components/Board.jsx` (game-over overlay)
- **Description:** When the game-over overlay appears, background elements (board cells, controls) remain focusable and clickable. The overlay does NOT implement a proper focus trap — it has a `onKeyDown` handler for Tab, but it doesn't prevent focus from escaping to elements behind it. Screen readers can still navigate to board cells behind the overlay.
- **Suggested fix:** Apply `inert` attribute to the board-container when game-over is true, or use `aria-hidden="true"` on the non-overlay content and properly trap focus within the overlay.

### FE-009: Loss chart canvas has hardcoded dark colors with potential contrast issues
- **Severity:** Low (accessibility)
- **Location:** `client/src/components/Dashboard.jsx` (`drawLossChart`)
- **Description:** Grid lines use `#333` stroke color on `var(--bg)` (#0d1117) background. The contrast ratio is approximately 1.4:1, which is below WCAG minimum of 3:1 for UI components. The min/max labels use `#888` on `#0d1117` (~4.5:1 — borderline OK). The chart data line uses `#e94560` which should be fine.
- **Suggested fix:** Increase grid line color to at least `#555` or `#666` for better contrast, or use dashed lines at lower opacity to maintain visual lightness while increasing luminance.

### FE-010: Toast notification overlaps interactive content on mobile
- **Severity:** Low (UX)
- **Location:** `client/src/index.css` (`.toast-notification`), `client/src/App.jsx`
- **Description:** The toast is `position: fixed; bottom: 20px; left: 50%`. On mobile viewports (`≤768px`), the game layout stacks vertically, and the toast can cover the controls or param panel buttons. No margin/padding is reserved for the toast.
- **Suggested fix:** Move the toast to `top: 20px` instead of `bottom: 20px`, or add bottom padding to the main content area when a toast is visible, or use a toast library that auto-positions without overlap.

### FE-011: WebSocket disconnect — no reconnecting indicator in menu view
- **Severity:** Low (UX)
- **Location:** `client/src/App.jsx` (menu render block)
- **Description:** In the menu view (`mode === 'menu'`), the connection status is shown as static text: "🟢 Połączono z serwerem" or "🔴 Brak połączenia". There's no visual indicator that the client is actively trying to reconnect (which it is — socket.io reconnection is configured). A user seeing "🔴 Brak połączenia" might assume the game is broken rather than waiting for reconnection.
- **Suggested fix:** Track `reconnectAttempts` state (already tracked!) and show "🔄 Ponowne łączenie (próba N)..." when disconnected but reconnecting, rather than just "Brak połączenia".

### FE-012: `Board` component `areEqual` compares `captures?.length` but not `captures` contents
- **Severity:** Low (potential rendering bug)
- **Location:** `client/src/components/Board.jsx` (`areEqual` function)
- **Description:** The memo comparator checks `prevProps.captures?.length !== nextProps.captures?.length` but not whether the captures array contents changed. If a new captures array has the same length but different positions, the board won't re-render for multi-capture animation. In practice, captures change rarely and always with the same length for a given path, but this is technically incorrect.
- **Suggested fix:** Compare captures contents: `JSON.stringify(prevProps.captures) !== JSON.stringify(nextProps.captures)` or compare each element.

### FE-013: No `<React.StrictMode>` wrapper in main.jsx
- **Severity:** Informational (development)
- **Location:** `client/src/main.jsx`
- **Description:** `main.jsx` renders `<ErrorBoundary><App /></ErrorBoundary>` without `React.StrictMode`. StrictMode helps catch unsafe lifecycles, deprecated APIs, and side-effect issues during development. This is not a bug but means some React best-practice warnings are suppressed.
- **Suggested fix:** Wrap in `<React.StrictMode>` for development builds (can be stripped in production).

### FE-014: `handleCellClick` reconnects to stale `connectedRef` — UI doesn't block moves during brief disconnect windows
- **Severity:** Low (edge case)
- **Location:** `client/src/App.jsx` (`handleCellClick`)
- **Description:** `handleCellClick` checks `connectedRef.current` and shows a toast if disconnected. This is good. However, there's no visual feedback on the board itself (no overlay, no disabled cursor) when disconnected. The board still looks fully interactive — cells highlight on hover, pieces look clickable. The only feedback is a toast that auto-dismisses after 3 seconds.
- **Suggested fix:** Add a visual "disconnected" indicator on the board area (e.g., a semi-transparent overlay with "Reconnecting..." text) when `connected` is false.

### FE-015: `PvAI` mode: selecting opponent's piece silently does nothing (no feedback)
- **Severity:** Low (UX)
- **Location:** `client/src/App.jsx` (`handleCellClick`, line ~310)
- **Description:** In PvAI mode, clicking on a black piece (AI's pieces) silently clears selection (`setSelected(null); setLegalMoves([])`). There's no audio cue, no toast, no visual shake. A user might think the click didn't register.
- **Suggested fix:** Show a subtle toast or animate the piece briefly when clicking an opponent's piece, e.g., "To pionki AI — wybierz swój pionek".
