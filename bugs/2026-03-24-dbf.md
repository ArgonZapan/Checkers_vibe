### BUG-DBF-001: `_replaceModel` race — disposed model exposed between lock release and assignment
**Severity:** Medium-High
**Location:** `server/ai/trainer.js:488-496` (`_replaceModel`)

**Description:**
`_replaceModel` acquires the model lock, disposes the old model, then **releases the lock before creating the new model**. The new model is created outside the lock. Between lock release and the caller's assignment (`this.modelWhite = await this._replaceModel(...)`), `this.modelWhite` still points to the disposed object. Any prediction that acquires the lock during this window will read a disposed model.

```js
async _replaceModel(old, ...createArgs) {
    const release = await this.acquireModelLock();
    try {
      if (old) { disposeModel(old); }  // old disposed
    } finally {
      release();  // lock released HERE
    }
    return createModel(...createArgs);  // new model created OUTSIDE lock
    // Window: this.modelWhite still points to disposed object
}
```

The prediction code in `server/index.js:203-209` reads `this.modelWhite` while holding the model lock, but doesn't detect disposed (non-null) objects:
```js
const model = turn === 1 ? trainer.modelWhite : trainer.modelBlack;
if (!model) return res.status(503)...;  // disposed model is NOT null — passes check
const result = await predict(model, ...);  // predict on disposed model → crash or garbage
```

**Reproduction:**
1. Start self-play (which trains and periodically recreates models via `_replaceModel`)
2. Simultaneously send an `/api/ai/predict` HTTP request
3. If predict acquires the lock after disposal but before assignment → predict on disposed model

**Fix:**
Keep the lock held for the entire dispose + create cycle, or create the new model *before* disposing the old one:
```js
async _replaceModel(old, ...createArgs) {
    const release = await this.acquireModelLock();
    try {
      const newModel = await createModel(...createArgs);  // create FIRST
      if (old) disposeModel(old);  // then dispose
      return newModel;
    } finally {
      release();
    }
}
```

### BUG-DBF-002: `_moveQueue` error recovery discards queued operations
**Severity:** Medium
**Location:** `server/index.js:638-644` (move handler) and `server/index.js:498-517` (startGame handler)

**Description:**
When a move fails, the error handler resets `socket._moveQueue = Promise.resolve()`. This breaks the serialization chain — any operations that were queued *after* the failed move in the same microtask queue may be lost, and the next operation starts with a clean promise chain regardless of whether the C++ engine is in a consistent state.

```js
socket._moveQueue = (socket._moveQueue || Promise.resolve())
  .then(() => handleMove(socket, data))
  .catch(err => {
    console.error('[WS] move error:', err.message);
    socket.emit('error', { message: 'Move failed' });
    socket._moveQueue = Promise.resolve();  // ← discards state
  });
```

If `handleMove` partially executed (e.g., player move succeeded via C++ but AI move failed), the C++ game state is now inconsistent with what the client expects, but the queue is reset and the next operation proceeds blindly.

**Reproduction:**
1. Start PvAI game
2. Make a player move that succeeds on C++ side
3. Kill C++ engine mid-AI-move → `aiMove` throws
4. Queue resets; next `startGame` proceeds without knowing C++ state is mid-game

**Fix:**
Instead of resetting to `Promise.resolve()`, either:
- Leave the queue in a rejected state so subsequent operations also fail until explicit recovery
- Or perform a C++ state consistency check before resetting

### BUG-DBF-003: Auto-save lock is not truly exclusive with HTTP reset
**Severity:** Low-Medium
**Location:** `server/index.js:1070-1096` (auto-save) vs `server/index.js:271-289` (HTTP reset) and `server/index.js:896-914` (WS reset)

**Description:**
The auto-save and HTTP/WS reset both use `acquireLock()` (same `_saveLock` promise chain), but the lock only serializes — it doesn't prevent the auto-save from *starting* immediately after a reset releases the lock. After `resetModel()` deletes model files from disk and creates fresh models, the auto-save interval fires and saves the fresh (untrained) models, overwriting any concurrent state. This is mostly benign but means there's a brief window where:

1. Reset acquires lock → resets model → deletes files → releases lock
2. Auto-save immediately acquires lock → sees `dirty=true` → saves fresh model to disk
3. Between steps, if the server crashes, the fresh-but-not-yet-saved state is lost

This is technically correct behavior (fresh state gets saved), but the `trainer.dirty = false` snapshot in auto-save (line 1085) happens *before* the actual save completes. If a new dirty event fires during the save, it's correctly caught next cycle — but if the save fails and `dirty = true` is restored in the catch block (line 1093), there's a subtle interaction where a concurrent `resetModel` could have already cleared the buffer/stats, and the "retry" save would persist empty state.

**Reproduction:**
1. Trigger auto-save (dirty=true, starts saving)
2. Immediately trigger WS `reset` (waits for lock)
3. Auto-save fails mid-save → restores `dirty = true`
4. Reset completes → buffer/stats/model cleared
5. Next auto-save cycle sees `dirty=true` → saves empty state (correct, but surprising)

**Fix:**
Add a generation counter that increments on reset. Auto-save should check the generation hasn't changed before restoring `dirty = true` on failure.

### BUG-DBF-004: `_replaceModel` fails silently if `createModel` throws
**Severity:** Medium
**Location:** `server/ai/trainer.js:488-496`

**Description:**
In `_replaceModel`, if `createModel()` throws after the old model has been disposed, the exception propagates to the caller. But the caller (e.g., `resetModel` line 547) assigns the result without a try/catch around the assignment:
```js
this.modelWhite = await this._replaceModel(this.modelWhite, { ...this.modelParams });
this.modelBlack = await this._replaceModel(this.modelBlack, { ...this.modelParams });
```
If `_replaceModel` for `modelWhite` succeeds but for `modelBlack` throws, `modelWhite` is new but `modelBlack` is whatever it was before (possibly disposed if the sequence of calls is different). The system is left in a partially-reset state with no recovery.

**Reproduction:**
1. Corrupt `this.modelParams` (e.g., invalid network size)
2. Call `resetModel()`
3. First `_replaceModel` succeeds, second throws
4. `modelWhite` is fresh, `modelBlack` is stale/disposed

**Fix:**
Wrap both replacements in a single try/catch, or make `_replaceModel` atomic (dispose only after create succeeds).

### BUG-DBF-005: Rate limit map cleanup interval runs even when empty
**Severity:** Cosmetic/Performance
**Location:** `server/index.js:62-74`

**Description:**
The `_rateLimitCleanupInterval` runs every `RATE_LIMIT_WINDOW_MS` (60s) regardless of whether the rate limit map has any entries. On a server with no traffic, this is a wasted 60s interval tick. Not a real bug, but unnecessary overhead.

**Fix:**
Skip cleanup if `_rateLimitMap.size === 0`.

### BUG-DBF-006: `setParams` handler modifies global CONFIG without synchronization
**Severity:** Low-Medium
**Location:** `server/index.js:753-800`

**Description:**
The `setParams` WebSocket handler directly mutates `CONFIG.server.speedMode`, `CONFIG.ai.strategy.white`, `CONFIG.ai.strategies.minimax`, etc. These are global mutable objects. If two clients in `aivai` mode send `setParams` concurrently (both pass the per-socket throttle but race on the global config), the final CONFIG state is unpredictable. The per-socket `_moveQueue` serialization doesn't protect global CONFIG mutations.

Additionally, reward weight updates (lines 790-800) iterate over `CONFIG.ai.strategy.white` and `CONFIG.ai.strategy.black` by name and re-freeze strategy objects. If both strategies point to the same config name, the second freeze overwrites the first — benign but confusing.

**Reproduction:**
1. Two browser tabs connect, both start aivai mode
2. Tab A sends `setParams({ speedMode: 'fast' })`
3. Tab B sends `setParams({ speedMode: 'normal' })` concurrently
4. Final speedMode depends on timing

**Fix:**
Add a global mutex or serialize `setParams` through a server-level queue (not per-socket).

### BUG-DBF-007: Self-play `stop()` is non-blocking — loop may run for seconds after stop
**Severity:** Low
**Location:** `server/ai/trainer.js:605-609` (`stop`) and `server/ai/trainer.js:691` (while loop)

**Description:**
`trainer.stop()` sets `this.running = false`, but the `_playGame` loop only checks this flag at the top of each iteration. The loop body has multiple `await` points (C++ fetches, model predictions, training). If `stop()` is called during a long-running operation (e.g., model training at line 1028-1035), the loop continues until the next check — potentially several seconds. During this time, `trainer.running` is `false` but the trainer is still actively modifying state.

The `start()` method guards with `if (this.running) return;`, so a concurrent `start()` won't launch a second loop. But `stop()` followed by rapid `start()` could result in two loops briefly overlapping if `stop()` hasn't taken effect yet.

**Reproduction:**
1. Start self-play, wait for it to be in the training phase
2. Call `stop()` → `running = false` but training continues
3. Immediately call `start()` → `running` is already false → sets `running = true` → launches new loop
4. Old loop's training finishes, checks `running` (now true), continues playing
5. Two `_playGame` calls run concurrently, both hitting C++ engine

**Fix:**
Make `stop()` await a "stopped" promise, or add a `_stopRequested` generation counter that `_playGame` checks at each await point.
